# Dev-Design 阶段开发人员交互模板

本文件包含 Dev-Design Skill 与开发人员交互的提案式模板。

## 核心交互原则（强制遵守）

**自主分析先行原则**：每个维度先自主分析，再决定是否需要用户交互。

1. **自主分析**：先读取 `architecture.md`，从中提取该维度对应的架构决策，再结合项目代码探索，自主形成开发方案提案
2. **闭环判断**：评估该维度是否能自主闭环
   - **能闭环**（架构已明确+项目代码可推断具体实现）→ 直接写入文档，标注 `[SELF-CLOSED]`，不询问用户
   - **需人为决策**（多种实现方式、架构留有选择空间）→ 形成带架构引用和推荐方案的提案
3. **提案交互**：仅在需人为决策时才交互，交互内容是**带架构决策引用 `[ARCH-REF]` 和推荐实现方案的提案**

**正确示例（提案式）**：
```text
[DEV-DESIGN-CLASS] 基于架构设计文档 [ARCH-REF: 3.3接口定义] 和项目代码探索，我提出以下核心类设计提案：

自主分析结论：
- 推荐类名：NotificationGroup [基于架构文档接口定义推断]
- 继承关系：继承 Parcelable [基于项目IPC传输需求推断]
- 推荐文件位置：frameworks/ans/core/notification_group.h [基于项目代码组织模式推断]

需确认的决策点：
1. 类名和继承关系是否符合预期？

如无异议，我将按此方案继续。
```

**禁止的错误示例（空洞提问）**：
```text
❌ 请确认核心类设计：
1. 类命名和结构？
2. 类关系？
（空洞提问，未附带架构引用和推荐方案）
```

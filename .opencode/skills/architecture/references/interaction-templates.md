# Architecture 阶段用户交互模板

本文件包含 Architecture Skill 与架构师交互的提案式模板。

## 核心交互原则（强制遵守）

**自主分析先行原则**：每个维度先自主分析，再决定是否需要用户交互。

1. **自主分析**：结合需求描述和项目代码探索，自主形成该维度的初步结论
2. **闭环判断**：评估该维度是否能自主闭环
   - **能闭环**（需求已明确、探索可推断）→ 直接写入文档，标注 `[SELF-CLOSED]`，不询问用户
   - **需人为决策**（多种方案、信息不足）→ 形成带具体方案的提案，向用户请求确认或选择
3. **提案交互**：仅在需人为决策时才交互，交互内容是**带分析结论和推荐方案的提案**，而非空洞提问
4. **维度聚焦**：每次交互聚焦单一维度，包含2-3个相关问题
5. **等待回复**：提出问题后，必须等待用户回复

**正确示例（提案式）**：
```text
[ARCHITECTURE-SCENARIO] 基于需求描述和项目代码分析，我形成以下使用场景提案：

自主分析结论：
- 用户痛点：<基于需求描述自主提炼>
- 发生频率：<基于项目现状推断>
- 当前临时方案：<基于代码探索发现的现有处理方式>

需确认的决策点：
1. 是否存在我分析未覆盖的其他痛点场景？
2. 发生频率推断是否准确？

如无异议，我将按此分析继续下一个维度。
```

**禁止的错误示例（空洞提问）**：
```text
❌ 请描述需求的使用场景：
1. 用户核心痛点是什么？
2. 问题发生的频率如何？
（空洞提问，未附带分析结论，让用户从零回答）
```

# 用户交互模板

本文件包含 Feature-Agent 和各 SubAgent 使用用户交互模板。

## 核心交互约束（强制遵守）

**必须遵守以下规则：**

1. **维度聚焦原则**：每次交互聚焦**单一维度**，包含该维度的2-3个相关问题
2. **问题相关性原则**：同一次交互的问题必须是相关的、有逻辑关联性的
3. **等待回复原则**：提出问题后，必须等待用户回复，不得继续抛出新问题
4. **逐步深入原则**：基于用户回复，再提出下一个维度的相关问题
5. **避免复合问题**：不得跨维度提问（如"功能细节和测试策略一起问"）

**正确示例（遵守维度聚焦原则）：**
```text
请描述使用场景：
1. 用户核心痛点是什么？
2. 问题发生的频率如何？
（这2个问题都属于"使用场景"维度，有逻辑关联）
```

**错误示例（违反维度聚焦原则）：**
```text
请回答：
1. 用户痛点是什么？（使用场景维度）
2. 不实现有什么影响？（业务价值维度）
3. 需要哪些权限？（约束维度）
（跨维度提问，违反单一维度聚焦原则）
```

**错误示例（问题过多）：**
```text
请描述：
1. 真实用户痛点是什么？
2. 发生频率如何？
3. 影响多少用户？
4. 有无替代方案？
5. 替代方案效果如何？
（一次交互包含5个问题，超出合理范围）
```

## 1. Architecture 阶段用户交互模板

### 交互 1: 使用场景分析

```text
[ARCHITECTURE-SCENARIO] 请描述需求的使用场景：

1. **用户核心痛点**: 这个功能解决的用户核心痛点是什么？
2. **发生频率**: 问题发生的频率如何？影响多少用户？

请简要回答。
```

### 交互 2: 业务价值评估

```text
[ARCHITECTURE-VALUE] 请评估需求的业务价值：

1. **不实现的影响**: 如果不实现这个功能，会有什么影响？
2. **实现的收益**: 实现后能带来什么价值？

请简要说明。
```

### 交互 3: 优先级确认

```text
[ARCHITECTURE-PRIORITY] 请确认需求的优先级：

1. **紧迫性**: 为什么现在要做这个需求？
2. **优先级判断**: P0/P1的依据是什么？

请明确优先级和理由。
```

### 交互 4: 依赖方分析

```text
[ARCHITECTURE-DEPENDENCY] 请分析需求的依赖关系：

1. **上游调用者**: 谁会调用这个功能？调用频率和场景？
2. **依赖服务**: 依赖哪些外部服务？稳定性如何？

请列出关键依赖。
```

### 交互 5: 影响方分析

```text
[ARCHITECTURE-IMPACT] 请分析需求的影响范围：

1. **下游模块**: 哪些模块会被影响？影响程度？
2. **系统影响**: 是否影响其他系统？需要协调哪些团队？

请说明主要影响点。
```

### 交互 6: 功能边界确认

```text
[ARCHITECTURE-BOUNDARY] 请明确功能边界：

1. **包含功能**: 本次要实现哪些功能？
2. **排除功能**: 明确不包含哪些功能？为什么？

请明确需求边界。
```

### 交互 7: 业务流程梳理

```text
[ARCHITECTURE-FLOW] 请描述业务流程：

1. **核心路径**: 正常场景的完整流程是什么？
2. **异常路径**: 各种异常情况如何处理？

请简要说明关键流程。
```

### 交互 8: 数据流向分析

```text
[ARCHITECTURE-DATA] 请描述数据流向：

1. **数据来源**: 输入数据从哪里来？格式和大小？
2. **数据去向**: 输出数据到哪里去？格式和用途？

请说明数据流转路径。
```

### 交互 9: 接口定义确认

```text
[ARCHITECTURE-API] 请定义接口：

1. **输入参数**: 参数名称、类型、必填性、默认值？
2. **返回值**: 返回数据结构、字段说明？

请提供关键接口定义。
```

### 交互 10: 技术方案评估

```text
[ARCHITECTURE-TECH] 请讨论技术方案：

1. **实现方式**: 有哪些可选方案？优缺点对比？
2. **技术风险**: 是否有技术难点或不确定性？

请说明技术实现思路。
```

### 交互 11: 性能要求确认

```text
[ARCHITECTURE-PERFORMANCE] 请明确性能要求：

1. **并发要求**: 预期的并发量？峰值场景？
2. **响应时间**: 预期的响应时间？超时处理？

请说明性能目标。
```

### 交互 12: 容错处理设计

```text
[ARCHITECTURE-FAULT] 请设计容错机制：

1. **失败策略**: 各种失败场景如何处理？
2. **降级方案**: 服务降级的触发条件和逻辑？

请说明容错方案。
```

### 交互 13: 权限控制确认

```text
[ARCHITECTURE-PERMISSION] 请确认权限要求：

1. **权限清单**: 需要哪些权限？权限来源？
2. **权限校验**: 校验时机和方式？

请说明权限控制方案。
```

### 交互 14: 参数校验规则

```text
[ARCHITECTURE-VALIDATION] 请定义参数校验：

1. **入参规则**: 参数格式要求、取值范围？
2. **边界值**: 最大值、最小值、边界场景处理？

请说明参数校验规则。
```

### 交互 15: 埋点打点设计

```text
[ARCHITECTURE-MONITOR] 请设计监控方案：

1. **上报指标**: 需要上报哪些指标？上报时机？
2. **监控项**: 需要监控哪些数据？监控阈值？

请说明监控和埋点方案。
```

### 交互 16: 兼容性要求

```text
[ARCHITECTURE-COMPAT] 请明确兼容性要求：

1. **版本兼容**: 是否兼容老版本？兼容范围？
2. **回滚机制**: 是否需要回滚？回滚方式？

请说明兼容性方案。
```

### 交互 17: 测试场景规划

```text
[ARCHITECTURE-TEST-SCENARIO] 请规划测试场景：

1. **正常场景**: 列举所有正常场景的测试点？
2. **异常场景**: 列举所有异常场景的测试点？

请列出关键测试场景。
```

### 交互 18: 验证方法确认

```text
[ARCHITECTURE-TEST-METHOD] 请确认验证方法：

1. **功能验证**: 如何验证功能正确实现？
2. **性能验证**: 如何验证性能达标？

请说明验证方法。
```

### 交互 19: 测试数据准备

```text
[ARCHITECTURE-TEST-DATA] 请准备测试数据：

1. **测试环境**: 需要什么样的测试环境？
2. **测试数据**: 需要准备哪些测试数据？

请说明测试数据方案。
```

### 交互 20: 需求名称确认

```text
[ARCHITECTURE-NAMING] 基于架构设计文档，建议需求名称:

- 推荐名称: <基于核心功能和架构特点自动生成>
- 命名理由: <为何选择此名称，体现核心价值>

请确认或自定义需求名称:
```

### Architecture 完成确认模板

```text
[ARCHITECTURE-COMPLETE] 已完成架构设计文档:

需求名称: <已确认的名称>

文档位置: .opencode/kb/features/${feature-name}/feature-architecture.md

包含维度:
✅ 1. 需求背景与价值 - 使用场景、业务价值、优先级
✅ 2. 上下游与边界 - 依赖方、影响方、明确边界
✅ 3. 功能细节 - 业务流程、数据流向、接口定义
✅ 4. 实现方案 - 技术方案、性能要求、容错处理
✅ 5. 约束与要求 - 权限控制、参数校验、埋点打点、兼容性
✅ 6. 测试策略 - 测试场景、验证方法、测试数据

请确认架构设计文档:
- **批准**: 进入 Dev-Design 阶段，开始开发设计
- **需要修改**: Architecture 子代理继续迭代，修改架构文档
- **拒绝**: 终止流程，等待用户提供新需求或重新讨论

请明确回复: [批准/需要修改/拒绝]
```
